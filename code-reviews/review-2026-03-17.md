# Code Review — 2026-03-17

## Architecture Summary
`neighborhood-meals` is a deliberately small full-stack teaching app with a Flask backend in `backend/` and a React/Vite frontend in `frontend/`, plus repo-level planning docs that are meant to preserve both a working happy path and intentional setup friction on `main`. The main entrypoints are `backend/app.py` for the API and `frontend/package.json` scripts for the UI; the key external dependencies are Python/Flask, Node/Vite, and the browser-to-backend HTTP boundary at `http://localhost:8000`. The main trust boundary is JSON submitted by the browser into the Flask routes, while the highest project risk right now is repo-shape drift: leftover scaffold metadata and partially stale docs can send users or future agents down paths that are different from the canonical backend/frontend workflow.

## Findings

| ID | Severity | Category | Location | Problem | Proposed Fix |
|----|----------|----------|----------|---------|--------------|
| R001 | High | Documentation / Correctness | `README.md:18-22` | The documented backend startup flow ends with `python app.py` even though the verified happy path requires the backend virtualenv interpreter or an activated venv. On a clean machine this command uses the system Python and immediately fails with `ModuleNotFoundError: No module named 'flask'`, so the reader-facing setup instructions are currently wrong for normal use. | Change the README to either activate the venv before running Python or invoke `./.venv/bin/python app.py` explicitly so the documented happy path matches the tested one. |
| R002 | Med | Architecture / Documentation | `pyproject.toml:5-35` | The repo root still advertises an installable Python package and console script from the init scaffold, but that metadata no longer matches the actual product shape. It declares `requires-python = ">=3.10"`, contradicting the canonical `>=3.11`, and the console script target `neighborhood-meals.main:main` is not a valid Python import path because module names cannot contain hyphens. This creates a dead, misleading setup path for tools and future agents. | Remove or repurpose the root scaffold package metadata so it either cleanly reflects the real repo role or stops pretending the repo root is an application package. At minimum, delete the broken console script and align version metadata with `product-definition.md`. |
| R003 | Low | Documentation | `AGENTS.md:84-100` | The document reference table says `WHERE_AM_I.md` should be read at every session start, but the startup protocol at the top of the same file omits it. That inconsistency weakens the handoff contract and makes it easier for future sessions to miss product-level orientation. | Update the startup protocol so it includes `WHERE_AM_I.md`, matching the document reference table and the repo's actual orientation workflow. |
| R004 | Low | Documentation | `AGENTS.md:94-95` | `architecture.md` is listed as the place to record technical decisions, and the review workflow expects it, but the file does not exist. That leaves technical decisions without a durable home and forces reviewers to infer architecture from code and planning docs. | Create `architecture.md` once the first cross-stack technical decisions are locked, or remove the references until the project is ready to maintain it. |
